# Technical Finding / Milestone

**Context**: Code review de Fase 0.5 (offline/resiliencia) — 2026-06-30 21:25

## The Discovery

El fallback offline de la app tiene una **doble captura** que deja el banner desconectado:

- Los wrappers de servidor (`src/server/sgc/index.ts`, `src/server/rondas/client.ts`, 27 usos de `safeConvexCall`) capturan el error offline y devuelven un fallback vacio en vez de propagarlo.
- Las paginas (`src/app/(protected)/sgc/page.tsx`, `src/app/(protected)/dashboard/data.ts`) tienen su propio `try/catch { if(!isConvexOffline) throw; backendOffline=true }`, pero como el error nunca sube, `backendOffline` queda `false` y `BackendOfflineBanner` no se renderiza.

Resultado: en offline la app rinde ceros silenciosos (sin crash, correcto para "no romper"), pero **no muestra banner**, por lo que el criterio "banner visible cuando Convex offline" de Target 2 y Target 3 no se cumple. `isConvexOffline` funciona en aislamiento; el problema es el cableado.

Fix recomendado: exponer el estado offline sin perder el fallback vacio (`safeConvexCallWithStatus` -> `{ data, offline }`, o `fetchQuery` crudo dentro del `try/catch` de pagina para las lecturas que alimentan banners).

Hallazgos secundarios: F2/F3 (misma raiz en `/dashboard/sgc` y subrutas), F4 (`isConvexOffline` inspecciona `cause` un solo nivel; deuda menor), F5 (14 mutaciones publicas en `convex/migrations.ts`; ya es Target 5).
