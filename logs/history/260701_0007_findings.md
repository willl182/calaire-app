# Technical Finding / Milestone

**Context**: Code review de Fase 2 (offline UX) del plan de recuperacion funcional de calaire-app.

## The Discovery

La Fase 2 migro correctamente SGC (resumen/documentos/normativa/mapa) y las rutas de ronda/participante a `safeConvexCallWithStatus` -> `{ data, offline }`, cerrando F1/F2/F3/F4. Pero el **antipatron de doble captura de F1 sigue vivo** en tres rutas que quedaron con `safeConvexCall` (traga el offline y devuelve fallback sin lanzar) envuelto en un `try/catch` que espera un throw:

- **F13 (Alta)** `/dashboard` admin: `loadAdminDashboardData` -> `backendOffline` nunca true -> banner muerto; muestra ceros silenciosos.
- **F14 (Media)** `/mi-dashboard`: try/catch SGC muerto + `EmptyParticipantState` ("sin rondas") se muestra en offline junto al banner (confunde offline con vacio).
- **F15 (Media)** `/dashboard/rondas/[id]`: `getRondaMetricasCompletas` con `safeConvexCall`; offline parcial -> metricas en cero sin banner.

Regla general: si una funcion server ya usa `safeConvexCall`, un `try/catch { isConvexOffline }` aguas arriba es codigo muerto. Para banner offline hay que usar la variante `*WithStatus` o `fetchQuery` crudo en el try/catch.

Bajos: F16 (forms de mutacion editables en offline en `.../[id]/sgc`), F17 (`BackendOfflineBanner` hardcodea `127.0.0.1:3212`/`ECONNREFUSED`), F18 (`/sgc/mapa` reexporta la pagina dashboard sin parametrizar).

Consecuencia: Target 2 cierra; **Target 3 permanece abierto hasta corregir F14-F15**.
