# Problem Record (Troubleshooting)

**Severity**: Medium

## The Problem

The `/dashboard/sgc` page initially drifted from the dashboard base layout and rendered wider than the other admin views. At desktop width, the SGC shell no longer matched `Participantes`, which made the page feel visually inconsistent even though the route still worked.

## User-Requested Corrections

- Make the SGC panel follow the exact same base layout as the other dashboard pages.
- Keep the width exactly the same as the other admin views.

## The Solution / Workaround

- Reused the same dashboard wrapper pattern instead of the old isolated SGC container.
- Moved the SGC header, KPI row, and tabs into the shared `max-w-7xl` content column.
- Added `min-w-0` and `w-full` to the SGC container and board sections so the table could scroll internally without expanding the outer shell.
- Measured the final rendered geometry in production with Playwright and confirmed it matched `Participantes`.

## Learnings to Preserve

- A visual width mismatch on a complex table view can come from the nested content layout, not just the outer page wrapper.
- When the user asks for "exactly the same base layout", verify with a rendered measurement, not only by matching class names.
- If a dashboard page contains wide tables, constrain the inner containers with `min-w-0` so the table scrolls instead of stretching the page.
