# Problem Record (Troubleshooting)

**Severity**: Medium

## The Problem

The agent-auth flow needed an email provider, but the only reliable value the user could provide immediately was the sender address. A production mailer also needs an API key and, depending on the provider, a verified sender/domain. Resend will not guarantee delivery with an unverified `from`.

There was also a local build wrinkle earlier in the session: Turbopack in the sandbox failed with a port/process error, but the code itself was valid.

## User-Requested Corrections

- Use the same email address as `MAIL_FROM` during development.
- Avoid blocking the flow on missing provider setup.
- Preserve a usable smoke-test path even when real email delivery is not configured.

## The Solution / Workaround

- Set `MAIL_FROM=wilsonsalasc@gmail.com` in local env and documentation.
- Keep the route responses returning `claim_view_url` and an `email_delivery` status so dev can proceed without live mail.
- Document that `RESEND_API_KEY` is still required for actual sending and that Resend expects a verified sender/domain for normal delivery.
- Verify the final app with `pnpm build` outside the sandbox.

## Learnings to Preserve

- For agent auth, treat delivery and claim state as separate concerns: the HTTP endpoint should stay usable even when mail delivery is skipped.
- If a provider requires sender verification, a personal mailbox is not automatically a production-safe `from` address.
- When the sandbox blocks Turbopack, use an elevated build run for a real production check instead of treating the sandbox failure as a code defect.
