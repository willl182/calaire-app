# Technical Finding / Milestone

**Context**: Consolidacion final de planes y evaluaciones SGC para CALAIRE-APP.

## The Discovery

Todas las evaluaciones en `grills/evals/` convergen en una direccion: no conviene elegir un plan completo tal como esta. La estrategia correcta es un hibrido con jerarquia:

1. `plan_oc.md` como base de implementacion inmediata.
2. `plan_codex.md` como fuente de reglas funcionales, visibilidad por rol, cronograma, cierre documental y evidencia serie/version.
3. `plan_pi.md` como arquitectura futura de matriz documental, procesos, casos SGC e importacion CSV.
4. `plan_agy.md` como resumen historico, no como especificacion implementable.

El documento final creado en `grills/evals/version_final_sgc.md` define que la Fase 1 debe ser cierre documental por ronda, no gestor documental global. La primera entrega exitosa debe permitir una transicion con trazabilidad:

```txt
activa -> documentacion_pendiente -> cerrada
```

sin abrir edicion a participantes, sin perder historial y sin presentar como oficiales codigos documentales no confirmados.

## Technical Context Preserved

- El repo actual solo tiene `rondas.estado = borrador | activa | cerrada`; agregar `documentacion_pendiente` exige revisar schema, tipos, dashboard, actions, formularios de participante y badges.
- `app/providers.tsx` usa `ConvexProvider`, no `ConvexProviderWithAuth`; SGC necesita una decision explicita de auth antes de exponer evidencias o mutaciones sensibles.
- La guia Convex local exige no aceptar `userId` como argumento de autorizacion; usar identidad derivada server-side con `ctx.auth.getUserIdentity()` cuando haya auth Convex real.
- La ruta Fase 1 recomendada es `/dashboard/rondas/[id]/sgc`, integrada en `RondaContextNav`.
- La matriz documental, importacion CSV, comentarios, notificaciones, casos SGC y F-PSEA-08 nativo quedan fuera de Fase 1.
