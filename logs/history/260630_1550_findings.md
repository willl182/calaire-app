# Finding Record (Solution)

**Date**: 2026-06-30 15:50 -05

## Context

Three runtime errors all sharing the same root cause were documented in `260630_1529_problems.md` and `260630_1530_problems.md`:

1. `listRondas` at `src/server/rondas/client.ts:511` throwing `TypeError: fetch failed (ECONNREFUSED 127.0.0.1:3212)`.
2. `loadAdminDashboardData` at `src/app/(protected)/dashboard/data.ts:29` propagating the same error from `Promise.all([listRondas(), listAllParticipantes(), listWorkOSUsers()])`.
3. `SgcPage` at `src/app/(protected)/sgc/page.tsx:12` propagating the same error from `Promise.all([listSgcMaestro(), listNormativaSgc(), listMapaSgc()])`.

The previous resolution (start `pnpm exec convex dev`) is correct but only treats the symptom. The user requested that the code itself degrade gracefully when the backend is offline.

## What Changed

- New helper `src/lib/convex-fallback.ts` exporting `isConvexOffline(error)` and `safeConvexCall(operation, fn, fallback)`.
- New tests `src/lib/convex-fallback.test.ts` (7 cases) covering detection of `ECONNREFUSED`, `fetch failed`, successful path, fallback path, and rethrow of unrelated errors.
- `listRondas` in `src/server/rondas/client.ts` now uses `safeConvexCall('listRondas', ..., [])` so an offline backend returns an empty array instead of throwing.
- `loadAdminDashboardData` in `src/app/(protected)/dashboard/data.ts` now wraps the whole `Promise.all` in try/catch. On `isConvexOffline`, it returns a degraded payload (`rondas: []`, `allParticipantes: []`, `workosUsers: []`, `backendOffline: true`). Other errors rethrow.
- `DashboardPage` in `src/app/(protected)/dashboard/page.tsx` renders an amber notice banner when `adminData?.backendOffline` is true.
- `SgcPage` in `src/app/(protected)/sgc/page.tsx` no longer constructs JSX inside try/catch (eslint react-hooks/error-boundaries). It now precomputes `documentos / normativa / mapa` in the data-loading try/catch and renders a single `SgcView` with the `backendOffline` flag.
- New route-level error boundaries:
  - `src/app/(protected)/dashboard/error.tsx`
  - `src/app/(protected)/sgc/error.tsx`
  
  Both reuse `isConvexOffline` to show a specific "Backend de Convex no disponible" message; non-offline errors keep the previous generic fallback.

## Behaviour After Fix

When the local Convex dev server is not running:

- `/dashboard` renders the dashboard shell with empty lists and an amber banner instructing the user to run `pnpm exec convex dev`.
- `/sgc` renders the SGC header and counters in zero, with the same amber banner.
- Any unhandled exception in those routes now hits the new `error.tsx` boundary instead of producing a raw `TypeError: fetch failed` page.

When Convex is online, the behaviour is identical to before (no extra notice).

## Learnings to Preserve

- `fetchQuery` from `convex/nextjs` throws `TypeError: fetch failed` whose `cause.code` is `ECONNREFUSED` when the local Convex deployment is offline. Detection should not rely on `instanceof TypeError` alone; check `cause.code`, the error message, and the `code` property on the error itself.
- ESLint's `react-hooks/error-boundaries` rule forbids JSX construction inside try/catch in server components. Always precompute data inside the try/catch and render JSX outside.
- The new helper is the canonical place to add future server-side `fetchQuery` wrappers. New routes should import `safeConvexCall` and `isConvexOffline` from `@/lib/convex-fallback` instead of duplicating the detection logic.
