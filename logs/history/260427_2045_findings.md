# Technical Finding — Fase 5: herramienta de migración Supabase → Convex

**Context**: Migración Supabase → Convex. Implementación de la herramienta operativa de migración de datos.

## The Discovery

La migración necesita preservar referencias entre UUIDs de Supabase y `Id<"...">` generados por Convex. Se resolvió con un script Node que importa en orden de dependencia y mantiene mapas en memoria:

1. `rondas`
2. `ronda_contaminantes`
3. `ronda_participantes`
4. `ronda_pt_items`
5. `ronda_pt_sample_groups`
6. `envios`
7. `envios_pt`
8. `fichas_registro`
9. subtablas de fichas

El script guarda el mapping final en `scripts/.migration-state.json` para auditoría.

### Seguridad operacional

- Por defecto aborta si Convex ya tiene datos en tablas migradas.
- `--dry-run` lee y cuenta Supabase sin escribir en Convex.
- `--wipe` limpia las tablas migradas en Convex antes de importar.
- La limpieza se encapsula en `convex/migrations.ts` y requiere `confirm: 'WIPE_CONVEX_MIGRATION_TABLES'`.

### Conversión de campos

- `timestamptz` → Unix ms con `new Date(value).getTime()`
- `date` → string ISO original
- `null`/string vacío opcional → `undefined`, para cumplir validators Convex
- `float[]` Supabase → `number[]`
- snake_case Supabase → camelCase Convex

## Key Files Affected

- `convex/migrations.ts` — queries/mutations de importación y limpieza
- `scripts/migrate-supabase-to-convex.mjs` — script operativo
- `package.json` — script `migrate:supabase-to-convex`
- `convex/_generated/api.d.ts` — regenerado con módulo `migrations`
