# Problem Record (Troubleshooting)

**Severity**: Medium

## The Problem

Two integration issues appeared during smoke testing:

1. `POST /agent/auth` initially returned a WorkOS redirect because AuthKit proxy intercepted the machine endpoint before the route handler.
2. After excluding all `/agent/*`, the human page `/agent/auth/claim/view` returned a server error because it calls `withAuth` outside AuthKit proxy coverage.
3. Resend returned a delivery failure because the configured sender domain is not verified, causing the route to return `502` and block the claim flow.

## User-Requested Corrections

- Continue after Resend setup.
- Keep the agent auth flow usable for smoke testing even if delivery is not yet production-ready.

## The Solution / Workaround

- Updated `proxy.ts` to exclude only the exact machine endpoints and keep `/agent/auth/claim/view` behind AuthKit.
- Changed `lib/mailer.ts` to return `email_delivery.status = "failed"` for provider-level Resend errors instead of throwing.
- Changed agent claim routes to catch unexpected mailer errors and still return `claim_token` + `claim_view_url` with failed delivery metadata.
- Configured `WORKOS_CLIENT_ID` in the local Convex deployment so `convex dev` could publish the new functions.

## Learnings to Preserve

- Do not use a broad `/agent/*` proxy exclusion when a nested page depends on `withAuth`.
- Resend API key presence is not equivalent to delivery readiness; the `from` domain must be verified.
- For this flow, claim creation and mail delivery are separate concerns. A delivery failure should be reported in `email_delivery`, not converted into a failed claim.
