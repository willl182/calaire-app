# Problem Record (Troubleshooting)

**Severity**: Medium

## The Problem

The admin "Reutilizar datos" action in the ficha editor was not restoring the full participant data section consistently. In some cases it appeared to only bring back part of the block, or it reused the ficha currently being edited instead of a previous template ficha.

## User-Requested Corrections

- Restore the full participant data block:
  - `nit_laboratorio`
  - `correo_laboratorio`
  - `nombre_laboratorio`
  - `nombre_responsable`
  - `cargo`
  - `ciudad`
  - `departamento`
  - `telefono`
- Do not copy lists or unrelated ficha sections.
- Use a previous ficha by NIT or email, not the active ficha itself.

## The Solution / Workaround

- Updated `convex/fichas.ts` so `findFichaTemplateByLookup` accepts an optional `excludeFichaId` and filters out the ficha being edited.
- Updated `app/(protected)/dashboard/rondas/[id]/participantes/[pid]/ficha/FichaAdminEditor.tsx` to pass the current ficha id into the lookup query.
- Kept the restore assignment limited to the participant data scalar fields only.

## Learnings to Preserve

- When reusing ficha data, always exclude the current editable ficha from template lookup results.
- If a restore appears incomplete, first verify whether the matched source document has null fields before assuming the UI is at fault.
- The participant data section is not the same as the lists section; do not reuse list state as part of the restore flow.
