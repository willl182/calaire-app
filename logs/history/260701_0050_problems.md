# Problem Record (Troubleshooting)

**Severity**: High (bloqueante)

## The Problem

F19 — Regresion de Fase 3: los guards de identidad nuevos rompen `rondas`/`pt`/`fichas` cuando Convex esta **online**.

`convex/rondas/*`, `convex/pt/index.ts` y `convex/fichas/index.ts` ahora dependen de `ctx.auth.getUserIdentity()`. Pero el cliente de servidor `src/server/rondas/client.ts` y `src/server/rondas/fichas.ts` invocan `fetchQuery(api..., args)` y `fetchMutation(api..., args)` **sin el tercer argumento `{ token }`**. El unico modulo que autentica es SGC (`src/server/sgc/index.ts` via `sgcToken()` -> `requireAuth().accessToken`).

Sin token, con un backend Convex real y un usuario autenticado, `getUserIdentity()` devuelve `null` y todo guard lanza `Autenticacion requerida`:
- Lecturas envueltas en `safeConvexCall` -> se traga el error -> fallback vacio silencioso (sin banner, porque no es offline).
- Mutaciones (`fetchMutation` directo) -> lanzan al usuario (crear ronda, `upsertEnvio`, `submitFinalPT`, `claimParticipanteToken`, mutaciones de ficha, etc.).

## Por que se colo

Toda la verificacion de Fase 3 (`pnpm build`, `pnpm test`, E2E) corrio con **Convex offline**: cada lectura degrada a fallback y las mutaciones no se ejercen contra un backend real. `convex/access.test.ts` solo prueba el **rechazo sin identidad**, nunca el camino autenticado. El defecto quedo 100% enmascarado.

## The Solution / Workaround

Fix propuesto (pendiente de implementar):
1. Extraer un helper `rondasToken()` analogo a `sgcToken()`: `(await requireAuth()).accessToken` (import desde `@/server/auth`).
2. Pasar `{ token }` como tercer argumento en cada `fetchQuery`/`fetchMutation` de `client.ts` y `fichas.ts` que ahora exige identidad.
3. Distinguir authz (401/no autorizado) de offline en lecturas con fallback, para no enmascarar rechazos como "offline".
4. Verificar con Convex online + sesion valida y añadir prueba `convex-test` con `withIdentity` (exito propio + rechazo cross-tenant).

## Learnings to Preserve

- **Un guard de identidad en el backend Convex no vale nada si el caller no reenvia el token.** En `convex/nextjs`, `fetchQuery`/`fetchMutation` requieren `{ token }` explicito; no hay token ambiental.
- **No declarar cerrada una fase de seguridad verificando solo con el backend offline.** El offline enmascara tanto la fuga (fallback vacio) como el fallo de authz. El DoD de seguridad exige una corrida online autenticada.
- Al agregar guards, revisar SIEMPRE los tres lados: funcion backend, wiring del cliente (token) y una prueba del camino autenticado, no solo del rechazo.
