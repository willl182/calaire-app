# Technical Finding: Fase 3 — Participantes orientados a pendientes

**Context**: UI/UX Redesign — Coordinator Dashboard Phase 3

## The Discovery

### Consolidación de queries para la vista de participantes

La página original usaba dos queries separadas con un join manual:
```ts
const [participantes, fichasMap] = await Promise.all([
  listParticipantes(rondaId),        // tipo RondaParticipante (viejo)
  listFichaResumenesByRonda(rondaId), // Record<string, FichaResumen>
])
```

Se reemplazó por una única query consolidada que ya contiene ficha_estado y envios_pt_count:
```ts
const participantes = await listParticipantesRondaResumen(rondaId)
```

Esto elimina el join manual y reduce las llamadas a Convex de 2 a 1.

### Filtros operativos persistentes en URL

Se implementó un sistema de filtrado server-side sin JavaScript del lado del cliente:
- El filtro activo se lee de `searchParams.filtro`
- `FiltroBar` es un Server Component que renderiza `<Link>` puros
- La función `filtrarParticipantes()` es pura y testeable sin infraestructura
- Los filtros con count 0 se muestran deshabilitados visualmente (no ocultos)

Valores de filtro: `todos | enlace_pendiente | ficha_pendiente | ficha_enviada | con_envios | sin_envios`

### `tiene_envio_final` derivado sin cambios Convex

En lugar de agregar un campo a la query Convex (que requeriría buscar `finalSubmittedAt` en `envioPt`), se derivó en el mapper:
```ts
const enviosPtCount = (p.envios_pt_count ?? 0) as number
// ...
tiene_envio_final: enviosPtCount > 0,
```

Esto es una aproximación conservadora. Si se necesita semántica estricta ("todas las celdas confirmadas"), habría que modificar `convex/rondas.ts`.

### Separación visual de acciones destructivas

Las acciones "Regenerar" y "Eliminar" ahora tienen estilos visualmente distintos:
- **Regenerar**: borde neutro, hover ámbar — acción reversible
- **Eliminar**: fondo rose-50, borde rose — acción destructiva

Antes ambas eran links inline sin separación visual.
