# Technical Finding — Fase 4: lib/ reescrita con Convex

**Context**: Migración Supabase → Convex. Reescritura de la capa de acceso a datos.

## The Discovery

`lib/rondas.ts` (1270 líneas, 27 funciones async) y `lib/fichas.ts` (269 líneas, 10 funciones async) fueron reescritos completamente para usar `fetchQuery`/`fetchMutation` de `convex/nextjs`. Las firmas públicas (tipos y parámetros) se mantuvieron 100% idénticas — ningún `app/**/actions.ts` requirió cambios.

### Problema de tipos generados

- Al iniciar, `convex/_generated/api.d.ts` tenía `ApiFromModules<{}>` (vacío) — Convex no había generado los tipos
- Solución: `CONVEX_TMPDIR=/home/w182/w421/calaire-app/.convex-tmp npx convex codegen`
  - Se necesitó crear `.convex-tmp/` manualmente: `mkdir -p /home/w182/w421/calaire-app/.convex-tmp`
  - Sin `CONVEX_TMPDIR`, el comando falla con error de filesystem cross-device

### Patron de mapping (camelCase Convex → snake_case lib)

Los tipos Convex usan camelCase (`rondaId`, `invitadoAt`, `workosUserId`) mientras los tipos del lib usan snake_case (`ronda_id`, `invitado_at`, `workos_user_id`). Se escribieron helpers privados por tabla:

```ts
function mapRondaWithContaminantes(r: any): Ronda
function mapParticipanteDoc(p: any): RondaParticipante
function mapEnvioDoc(e: any): Envio
function mapPTItemDoc(item: any): RondaPTItem
function mapPTSampleGroupDoc(g: any): RondaPTSampleGroup
function mapParticipantePTDoc(p: any): RondaParticipantePT
function mapEnvioPTDoc(e: any): EnvioPT
// fichas:
function mapFichaDoc(doc: any): FichaRegistro
function mapAcompananteDoc(a: any): Acompanante
function mapAnalizadorDoc(a: any): Analizador
function mapInstrumentoDoc(i: any): Instrumento
function mapFichaResumenFromResult(r: any): FichaResumen
```

Se usan `any` con comentarios `// eslint-disable-next-line` porque las funciones Convex retornan tipos inferidos de schema que no están directamente exportados.

### upsertFichaScalars: doble capa de mapping

La función lib recibe campos en snake_case (`nombre_laboratorio`, `dec_datos_correctos`, etc.) pero el mutation Convex espera camelCase (`nombreLaboratorio`, `decDatosCorrectos`). Solución:

```ts
const SCALAR_FIELD_MAP: Record<FichaScalarField, ConvexField> = {
  nombre_laboratorio: 'nombreLaboratorio',
  dec_datos_correctos: 'decDatosCorrectos',
  // ...15 campos total
}
```

Además, el mutation Convex distingue `valueString` y `valueBoolean` como args separados (no un `value: string | boolean`). El lib function hace el dispatch:
```ts
typeof value === 'boolean' ? { valueBoolean: value } : { valueString: value }
```

### upsertEnvioPT: mutation + query posterior

El mutation Convex retorna solo el `Id`. La función lib debe retornar `EnvioPT` completo. Solución: tras el mutation, llamar a `fetchQuery(api.pt.getEnvioPT, {...})`.

### getOrCreateFicha: getFichaById añadido a convex/fichas.ts

El mutation `getOrCreateFicha` retorna `Id<'fichasRegistro'>`. Para construir `FichaRegistro` completo se añadió una query `getFichaById` a `convex/fichas.ts` que hace `ctx.db.get(fichaId)`.

### listAllParticipantes: cast de estado

La función Convex retorna `estado: string` (no tipado como `EstadoRonda`). Se necesitó un `.map()` con `estado: ronda.estado as EstadoRonda` en la función lib.
