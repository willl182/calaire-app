# Problem Record (Troubleshooting)

**Severity**: High

## The Problem

The production SGC page at `/dashboard/rondas/kd77ck9jqbeafg5g61c7cw0vrh8756qr/sgc` failed after authentication work with a Convex error:

```txt
{"code":"NoAuthProvider","message":"No auth provider found matching the given token"}
```

Before that, the page also failed with:

```txt
No hay sesion activa para operar SGC.
```

## User-Requested Corrections

- Deploy the fix to production.
- Diagnose why the page was broken.
- Save the session state after the fix.

## The Solution / Workaround

- `lib/sgc/index.ts` stopped reading the `workos-access-token` cookie directly in server components.
- `sgcToken()` now uses `requireAuth()` and `auth.accessToken`.
- `convex/auth.config.ts` was corrected so the WorkOS user management provider no longer declares `applicationID` on the `https://api.workos.com/user_management/${WORKOS_CLIENT_ID}` issuer.
- Convex production was redeployed after the auth config fix.

## Learnings to Preserve

- In AuthKit on Next.js, the server-side access token should come from `withAuth()`/`requireAuth()`, not from a manually read cookie in server components.
- For Convex WorkOS auth config, the user management issuer needs to match the JWT issuer exactly; an extra `applicationID` on the second provider can produce `NoAuthProvider`.
- When diagnosing production auth failures, check Vercel request logs first because they expose the exact thrown server-side error.
